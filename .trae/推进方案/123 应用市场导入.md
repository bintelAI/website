# 应用市场模板导入到团队应用推进方案

## 1. 背景与目标

当前应用市场已有模板导入入口，但导入能力存在缺口：用户从应用市场选择模板后，无法稳定、完整地把模板对应的应用结构导入到自己的团队应用中。目标是实现“除业务数据外，应用运行所需的配置与结构全部导入”，让团队可以直接基于市场模板二次配置和使用。

本方案聚焦应用市场中的应用、模板、AI 工作流等资源导入到目标团队应用，覆盖菜单、表格、视图、字段、审批、工作流、角色、权限、页面配置、资源依赖等内容。业务数据、用户私有数据、运行日志、审批实例、工作流执行记录等不随模板导入。

当前仓库中可确认的现状：

- `app/pages/solutions.vue` 已展示应用市场资源列表，资源类型包含 `application`、`flow_plugin`、`template`。
- `server/api/open/market/resource/list.get.ts` 已代理市场资源列表接口 `/open/market/resource/list`。
- 官网侧“导入”按钮当前跳转到 `https://dimens.bintelai.com/#/market`，未承载完整导入闭环。
- 真正的团队应用导入逻辑应在 Dimens 主应用或后端服务中完成，本文作为需求和推进方案沉淀。

## 2. 核心结论

模板导入不能只复制单个表格或单个工作流，而应按“应用包”导入。应用包需要描述完整依赖图，并由导入服务按依赖顺序完成校验、复制、ID 重映射、引用修复、权限挂载和导入结果回写。

推荐导入边界：

- 导入：应用元信息、菜单、页面、表格结构、字段、视图、表单、仪表盘、审批定义、工作流定义、角色定义、权限规则、自动化规则、资源引用、默认配置。
- 不导入：业务行数据、附件业务内容、审批实例、工作流运行记录、评论、操作日志、团队成员、用户账号、真实外部密钥、支付订单、消息通知历史。
- 可选导入：示例数据。默认关闭，仅用于演示模板，且必须标记为 `sampleData`，不能混入真实业务数据。

## 3. 用户场景

### 3.1 模板使用方

团队管理员或有应用创建权限的成员，在应用市场中选择模板，点击导入，选择目标团队和导入方式，系统自动创建一个新的团队应用，并带入完整应用配置。

### 3.2 模板发布方

模板作者将已有应用发布到市场时，系统导出应用结构与配置，过滤业务数据，生成可版本化的模板包。模板包需要能被预检、安装、升级和回滚。

### 3.3 平台管理员

平台管理员需要能排查导入失败原因，查看导入任务状态、导入日志、资源映射关系和异常数据。

## 4. 范围定义

### 4.1 必须导入的内容

| 模块 | 导入内容 | 说明 |
| --- | --- | --- |
| 应用基础信息 | 应用名称、描述、图标、封面、分类、版本、模板来源 | 导入后可重命名，保留来源模板 ID 和版本 |
| 菜单 | 菜单树、排序、图标、路由、入口页面、显隐规则 | 菜单引用页面、表格、报表、工作流时需要重映射 |
| 页面 | 页面布局、组件配置、页面变量、数据源绑定、操作按钮 | 不导入页面产生的业务数据 |
| 表格 | 表、字段、字段类型、字段配置、索引、关联关系、公式、默认值 | 不导入表格行数据 |
| 视图 | 表格视图、看板、甘特、日历、筛选、排序、分组、列宽、条件样式 | 视图中的字段 ID 需要重映射 |
| 表单 | 新增/编辑表单、字段布局、校验规则、默认值、提交后动作 | 表单提交记录不导入 |
| 审批定义 | 审批模板、节点、条件分支、会签/或签规则、抄送、催办配置 | 审批实例、审批记录不导入 |
| AI 工作流 | 工作流画布、节点、连线、变量、触发器、输入输出 schema、发布状态 | 执行记录和运行上下文不导入 |
| 自动化规则 | 触发条件、执行动作、定时规则、Webhook 配置占位 | 外部密钥不导入，只导入配置占位 |
| 角色 | 模板内置角色、角色说明、角色与权限绑定关系 | 不导入具体成员 |
| 权限 | 应用权限、菜单权限、表权限、字段权限、视图权限、操作权限、工作流执行权限 | 主体 ID 需从模板角色映射到目标团队角色 |
| 资源依赖 | 图标、封面、组件资源、插件依赖、市场插件版本 | 需校验依赖是否存在或自动安装 |

### 4.2 明确不导入的内容

- 表格业务行数据、真实客户数据、订单数据、项目数据等。
- 附件业务内容。模板封面、图标等静态素材可以导入。
- 审批实例、审批意见、审批附件、审批流转历史。
- 工作流执行记录、节点运行日志、运行变量快照。
- 操作日志、访问日志、审计日志。
- 团队成员、用户账号、部门组织架构。
- 第三方系统真实密钥、Token、Cookie、数据库连接密码。
- 计费订单、支付记录、套餐权益变更记录。

## 5. 导入对象模型

建议统一抽象为模板包 `TemplatePackage`：

```json
{
  "packageId": "market-template-id",
  "version": "1.0.0",
  "resourceType": "application",
  "sourceTeamId": "template-team",
  "manifest": {},
  "entities": [],
  "dependencies": [],
  "permissions": [],
  "checksum": "sha256"
}
```

核心实体建议包含：

- `app`：应用基础信息。
- `menu`：菜单树与入口。
- `page`：页面布局与组件。
- `table`：表结构。
- `field`：字段定义。
- `view`：视图定义。
- `form`：表单定义。
- `approvalDefinition`：审批定义。
- `workflowDefinition`：工作流定义。
- `automationRule`：自动化规则。
- `roleDefinition`：模板角色。
- `permissionPolicy`：权限规则。
- `asset`：模板静态资源。
- `pluginDependency`：插件和扩展依赖。

每个实体必须包含稳定的模板内 ID，例如 `sourceId`，导入后生成目标团队的新 ID，并记录 `sourceId -> targetId` 映射。

## 6. 导入流程

### 6.1 市场侧导出流程

1. 用户选择已有应用发布为模板。
2. 系统执行导出预检，确认应用结构完整、引用关系可解析。
3. 导出应用结构和配置，剔除业务数据。
4. 生成模板包 manifest、依赖图、版本号和校验摘要。
5. 写入市场资源，状态为草稿或待审核。
6. 审核通过后，市场资源可被目标团队导入。

### 6.2 团队侧导入流程

1. 用户在应用市场点击导入。
2. 系统校验用户是否具备目标团队的应用创建权限。
3. 用户选择目标团队、应用名称、冲突处理方式和是否导入示例数据。
4. 后端创建导入任务，返回任务 ID。
5. 导入服务拉取模板包并校验版本、签名、完整性和依赖。
6. 解析依赖图，按顺序导入资源。
7. 为所有实体生成新 ID，写入映射表。
8. 修复菜单、页面、视图、审批、工作流、权限中的引用 ID。
9. 挂载角色和权限，默认不绑定具体成员。
10. 完成后返回新应用 ID，并记录导入日志。
11. 如果失败，按任务事务策略回滚已创建资源，保留失败日志。

### 6.3 推荐依赖顺序

1. 应用基础信息。
2. 静态资源和插件依赖。
3. 角色定义。
4. 表结构和字段。
5. 视图、表单、页面。
6. AI 工作流定义。
7. 审批定义和自动化规则。
8. 菜单树。
9. 权限策略。
10. 应用发布状态和默认入口。

## 7. 关键机制设计

### 7.1 ID 重映射机制

导入过程中所有模板内实体都不能复用源 ID。系统需要生成目标团队内的新 ID，并维护映射：

| 类型 | 源 ID | 目标 ID | 说明 |
| --- | --- | --- | --- |
| table | sourceTableId | targetTableId | 字段、视图、权限引用需要替换 |
| field | sourceFieldId | targetFieldId | 公式、筛选、审批条件需要替换 |
| workflow | sourceWorkflowId | targetWorkflowId | 菜单、按钮、自动化动作需要替换 |
| role | sourceRoleId | targetRoleId | 权限主体需要替换 |
| page | sourcePageId | targetPageId | 菜单入口需要替换 |

映射表既用于导入过程，也用于排查问题和未来模板升级。

### 7.2 引用修复机制

以下配置必须进行引用修复：

- 菜单路由中的应用 ID、页面 ID、表 ID、工作流 ID。
- 页面组件数据源绑定的表 ID、视图 ID、字段 ID。
- 视图筛选、排序、分组、条件格式中的字段 ID。
- 公式字段中的字段引用。
- 关联字段、查找字段中的目标表和目标字段。
- 审批节点条件中的表字段、角色、工作流动作。
- 工作流节点中的数据表、字段、页面、审批、Webhook、变量引用。
- 权限规则中的主体 ID、资源 ID、动作 ID。

### 7.3 菜单导入机制

菜单应以树结构导入，保留层级、排序、图标、显示条件和入口类型。菜单节点引用资源时不应保存旧链接，而应使用资源类型加源 ID，在导入后统一替换为目标 ID。

菜单导入验收重点：

- 一级、二级、多级菜单顺序一致。
- 菜单入口能打开目标团队中新创建的页面、表格、报表或工作流。
- 权限控制后，无权限成员不可见不可访问。

### 7.4 表格导入机制

表格导入只导入结构和配置，不导入业务行数据。需要覆盖：

- 字段类型、字段名称、字段说明、必填、唯一、默认值。
- 单选、多选选项。
- 公式、汇总、关联、查找字段。
- 字段权限、字段显隐、字段校验。
- 表之间的关联关系。
- 视图配置、筛选排序、分组、条件样式。

复杂字段处理：

- 关联字段需要等目标表全部创建后再修复引用。
- 公式字段需要替换公式内字段标识。
- 成员字段不能带入模板源团队成员，只保留“成员字段”配置。
- 附件字段只保留字段结构，不导入附件业务文件。

### 7.5 审批导入机制

审批定义应作为配置导入，不导入审批实例。需要覆盖：

- 审批名称、适用表单、触发条件。
- 节点类型：发起人、审批人、抄送人、条件分支、结束节点。
- 审批方式：会签、或签、顺序审批。
- 节点候选人规则：角色、字段成员、发起人上级、固定人员占位。
- 审批动作：更新字段、触发工作流、发送通知。

成员相关处理：

- 模板中固定人员不能直接导入为目标团队成员。
- 固定人员应转换为“待配置占位”或映射到目标团队角色。
- 导入完成后，如果存在待配置审批人，应用应进入“需配置”状态，不允许直接启用对应审批流。

### 7.6 AI 工作流导入机制

工作流应导入定义和编排，不导入执行记录。需要覆盖：

- 画布节点、连线、节点配置。
- 输入输出参数、变量、上下文 schema。
- 触发器：手动、表格事件、定时任务、Webhook。
- 模型节点、工具节点、HTTP 节点、数据库节点、审批节点。
- 发布状态和启用状态。

安全处理：

- API Key、Token、数据库密码等敏感配置不导入。
- 外部连接器导入为“未配置”状态。
- 工作流存在未配置密钥时不可自动启用。

### 7.7 角色导入机制

角色导入只导入角色定义，不导入成员关系。需要支持：

- 创建模板内置角色，例如“应用管理员”“审批负责人”“业务成员”“只读成员”。
- 如果目标团队已有同名角色，按导入策略处理：复用、重命名或新建。
- 角色权限随角色导入，但成员绑定由团队管理员后续配置。

推荐默认策略：

- 应用级角色优先新建到目标应用内，避免污染团队全局角色。
- 团队级角色仅在模板明确声明时创建。
- 同名冲突默认追加模板名称后缀，避免覆盖已有角色。

### 7.8 权限导入机制

权限必须跟随资源和角色完成重映射。需要覆盖：

- 应用访问权限。
- 菜单可见权限。
- 表格查看、新增、编辑、删除、导出权限。
- 字段查看、编辑权限。
- 视图访问权限。
- 审批发起、审批处理、审批管理权限。
- 工作流查看、编辑、执行、管理权限。
- 页面按钮和操作权限。

权限原则：

- 不把模板源团队成员导入为权限主体。
- 只导入基于角色、资源、动作的策略。
- 导入完成后，创建者默认拥有新应用管理员权限。
- 对未映射主体的权限规则标记为待配置，不静默丢弃。

## 8. 冲突处理策略

导入时可能存在应用名、角色名、插件版本、资源依赖冲突。建议提供三种策略：

| 策略 | 适用场景 | 行为 |
| --- | --- | --- |
| 新建副本 | 默认导入 | 所有资源新建，名称冲突自动追加后缀 |
| 智能复用 | 高级导入 | 可复用目标团队已有插件、角色、连接器 |
| 覆盖升级 | 模板升级 | 仅对同一来源模板的历史导入应用执行升级 |

第一期建议只实现“新建副本”，降低破坏性。

## 9. 导入任务与回滚

导入应采用异步任务，避免大模板超时。

任务状态：

- `pending`：等待执行。
- `validating`：校验模板包。
- `importing`：导入中。
- `repairing`：引用修复中。
- `completed`：导入完成。
- `failed`：导入失败。
- `rolled_back`：已回滚。

回滚要求：

- 同一次导入创建的资源需要标记 `importTaskId`。
- 导入失败时优先自动删除本次任务创建的资源。
- 若删除动作受现有约束限制，则将资源标记为不可见和待清理。
- 保留导入日志和失败原因，便于排查。

## 10. 接口需求

### 10.1 市场资源列表

已有代理接口：

- `GET /api/open/market/resource/list`

需要补充能力：

- 返回资源是否支持导入。
- 返回模板版本、依赖摘要、资源类型、导入入口。

### 10.2 模板详情

建议新增：

- `GET /open/market/resource/{resourceId}`

返回模板详情、版本、预览信息、导入说明、依赖清单。

### 10.3 导入预检

建议新增：

- `POST /open/market/resource/{resourceId}/import/preview`

入参：

- `targetTeamId`
- `targetProjectId`
- `importMode`
- `withSampleData`

返回：

- 将创建的资源数量。
- 依赖是否满足。
- 冲突列表。
- 需要用户补充配置的项。
- 风险提示。

### 10.4 创建导入任务

建议新增：

- `POST /open/market/resource/{resourceId}/import`

入参：

- `targetTeamId`
- `targetProjectId`
- `appName`
- `importMode`
- `roleConflictStrategy`
- `withSampleData`

返回：

- `importTaskId`
- `status`
- `targetAppId`

### 10.5 查询导入任务

建议新增：

- `GET /open/market/import/tasks/{taskId}`

返回：

- 当前状态。
- 进度百分比。
- 已创建资源。
- 失败原因。
- 待配置项。
- 新应用入口。

## 11. 权限与安全要求

- 发起导入的用户必须具备目标团队的应用创建权限。
- 市场资源必须处于可导入状态，且用户有可见权限。
- 私有模板只能被授权团队或用户导入。
- 模板包必须做完整性校验，避免被篡改。
- 导入过程中不允许写入源团队 ID、源用户 ID、源密钥。
- 敏感字段必须脱敏或变为配置占位。
- 所有导入操作需要记录审计日志。

## 12. 产品交互需求

### 12.1 应用市场列表

- 每个资源展示类型：应用、模板、AI 工作流。
- 支持查看详情、预览、导入。
- 导入按钮需要根据用户权限展示可用、禁用或登录提示。

### 12.2 导入弹窗

导入弹窗需要包含：

- 目标团队选择。
- 应用名称输入。
- 导入模式选择，第一期默认新建副本。
- 是否导入示例数据，默认关闭。
- 预检结果展示。
- 需要后续配置的风险项展示。

### 12.3 导入进度

- 展示导入阶段和进度。
- 失败时展示明确原因和重试入口。
- 成功后展示“进入应用”和“继续配置权限”。

### 12.4 导入后配置提醒

如存在以下情况，应用进入“需配置”状态：

- 审批人未映射。
- 外部连接器密钥未配置。
- 工作流触发器未启用。
- 权限主体存在待绑定角色。

## 13. 非功能需求

- 性能：中型模板导入应在 30 秒内完成；大型模板采用异步任务，页面可轮询进度。
- 幂等：同一导入任务重复执行不能产生重复资源。
- 可观测：导入日志可定位到实体级失败原因。
- 兼容：模板包需要携带版本，支持后续 schema 升级。
- 可恢复：失败任务可以重试或回滚。
- 简洁：第一期优先实现新建导入，不做复杂覆盖升级。

## 14. 分阶段推进

### 第一阶段：补齐应用级完整导入

目标：实现从市场模板新建团队应用，导入除业务数据外的核心配置。

范围：

- 模板包 manifest。
- 应用、菜单、表格、字段、视图、页面。
- 角色定义和权限策略。
- 导入任务、ID 映射、引用修复、失败回滚。

验收：

- 用户可从市场选择模板并导入为目标团队的新应用。
- 导入后菜单、表格、视图、页面可正常访问。
- 表格无业务行数据。
- 创建者具备新应用管理员权限。

### 第二阶段：补齐审批与工作流

目标：让审批定义和 AI 工作流随应用模板完整迁移。

范围：

- 审批定义导入。
- 工作流定义导入。
- 自动化规则导入。
- 外部连接器和敏感配置占位。
- 未配置项提醒。

验收：

- 审批模板可进入目标应用并完成节点引用修复。
- 工作流画布、节点和触发器配置导入完整。
- 密钥缺失时工作流不会自动启用，并提示管理员配置。

### 第三阶段：模板升级与生态增强

目标：支持已导入模板的版本升级和更复杂的市场生态。

范围：

- 模板版本比较。
- 同来源应用升级。
- 插件依赖自动安装或提示安装。
- 导入报告和差异预览。

验收：

- 已导入应用可识别来源模板版本。
- 管理员可查看升级差异并选择是否升级。
- 升级不覆盖团队已产生的业务数据。

## 15. 验收标准

### 15.1 完整导入

Given 用户拥有目标团队应用创建权限，When 从市场导入一个包含菜单、表格、页面、角色、权限的应用模板，Then 系统创建新应用，并完整导入除业务数据外的所有配置。

### 15.2 表格不带业务数据

Given 模板源应用表格中存在业务行数据，When 用户导入模板，Then 目标应用只包含表结构、字段和视图，不包含源业务行数据。

### 15.3 权限重映射

Given 模板中存在角色和权限策略，When 导入到目标团队，Then 系统创建或复用目标角色，并将权限策略绑定到目标资源 ID，不引用源团队成员。

### 15.4 审批定义导入

Given 模板中存在审批流程，When 导入完成，Then 目标应用中存在审批定义，审批节点引用的表、字段、角色均已替换为目标资源；固定人员类节点进入待配置状态。

### 15.5 工作流导入

Given 模板中存在 AI 工作流，When 导入完成，Then 目标应用中存在工作流定义，节点和连线完整；涉及密钥的节点为未配置状态，不自动运行。

### 15.6 失败回滚

Given 导入过程中某个资源创建失败，When 导入任务结束，Then 本次任务已创建资源被回滚或标记不可见，并保留失败日志。

## 16. 待确认问题

- 市场模板包当前是否已有 manifest 或仅存储资源记录。
- Dimens 主应用中菜单、表格、审批、工作流、权限的真实数据表结构。
- 权限模型是团队级 RBAC、应用级 RBAC，还是资源级 ACL 与 RBAC 混合。
- 审批节点中“上级”“部门”等组织架构规则是否已经存在。
- 工作流节点中外部连接器、模型配置、MCP 工具配置的存储方式。
- 模板是否允许携带示例数据，若允许，需要定义示例数据标识和隔离规则。
- 当前导入功能的缺陷清单和线上失败案例。

## 17. 研发拆解建议

1. 先梳理真实数据模型，列出所有可导入实体和引用字段。
2. 设计模板包 schema，建立实体类型、依赖关系和版本规则。
3. 实现导出服务，确保业务数据过滤可靠。
4. 实现导入预检，提前暴露冲突、依赖和权限问题。
5. 实现导入任务、事务、ID 映射和引用修复。
6. 接入菜单、表格、页面导入。
7. 接入角色和权限导入。
8. 接入审批和工作流导入。
9. 补齐产品交互、导入进度、失败日志和导入后配置提醒。
10. 用典型模板做端到端回归：项目管理、财务审批、营销管理、售后服务。

## 18. 风险点

- 只复制资源不修复引用，会导致菜单打不开、视图筛选失效、工作流节点找不到表。
- 权限主体如果直接复制源用户 ID，会造成越权或不可用。
- 审批和工作流中的密钥如果直接导入，会造成安全风险。
- 表格关联、公式、查找字段如果处理不完整，会造成应用结构看似导入成功但运行失败。
- 缺少导入任务和回滚机制时，失败会留下半成品应用。
- 模板升级如果没有来源版本和映射关系，后续无法安全演进。

## 19. 第一版最小可交付范围

第一版建议收敛为：

- 从市场导入一个应用模板到目标团队。
- 新建应用副本，不覆盖已有应用。
- 导入应用基础信息、菜单、页面、表格、字段、视图、角色、权限。
- 不导入业务数据。
- 完成 ID 重映射和引用修复。
- 创建者自动成为应用管理员。
- 支持导入任务状态和失败回滚。

审批、AI 工作流、自动化规则可以在第一版保留 schema 和占位，第二阶段补齐完整执行能力。
